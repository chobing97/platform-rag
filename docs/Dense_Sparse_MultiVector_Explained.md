# Dense, Sparse, Multi-vector 검색 상세 설명

**작성일:** 2026.03.06
**맥락:** Knowledge Lake 아키텍처의 Hybrid RAG 검색 엔진에서 사용하는 세 가지 검색 방식

---

## 1. Dense Retrieval (밀집 벡터 검색)

### 원리

문장 전체의 **의미**를 하나의 고정 길이 벡터(숫자 배열)로 압축한다.

```
입력: "IBKR이 DAOL을 Execution-Only로 분류했다"
                    │
                    ▼
              [ 0.023, -0.147, 0.891, 0.034, ... , -0.562 ]
                         1,024개의 숫자 (bge-m3 기준)
```

두 문장의 벡터 간 **코사인 유사도**를 계산해서 의미적으로 가까운 문서를 찾는다.

```
질문:  "clearing broker 관련 계약 조항이 뭐야?"
        → 벡터 A: [0.034, -0.112, 0.773, ...]

문서1: "ECACA §7.A에서 Clearing Broker의 역할을 정의한다"
        → 벡터 B: [0.041, -0.098, 0.801, ...]

문서2: "마케팅 예산안을 첨부합니다"
        → 벡터 C: [-0.567, 0.234, 0.012, ...]

유사도: A↔B = 0.94 (높음 → 관련 있음 ✓)
유사도: A↔C = 0.11 (낮음 → 관련 없음 ✗)
```

### 강점

| 강점 | 예시 |
|---|---|
| **동의어/유의어 인식** | "해고" 검색 → "인력 감축", "구조조정" 문서도 찾음 |
| **다국어 교차 검색** | "clearing broker" 검색 → "청산 브로커" 포함 한국어 문서도 찾음 |
| **의도 파악** | "IBKR이 우리를 어떻게 보고 있어?" → 분류/평가 관련 문서를 찾음 |

### 약점

| 약점 | 예시 |
|---|---|
| **정확한 용어 매칭 실패** | "§7.A" 검색 → 조항번호가 벡터에 정확히 인코딩 안 됨. 엉뚱한 조항 반환 가능 |
| **고유명사/코드 검색 약함** | "DEC-2026-0001" → 의미가 아닌 식별자라 벡터로 구분 어려움 |
| **짧은 키워드 검색 부정확** | "microcap" 한 단어 → 문맥 부족으로 관련 없는 문서 섞임 |

## 2. Sparse Retrieval (희소 벡터 검색)

### 원리

문장을 **단어(토큰) 단위의 중요도 가중치**로 표현한다. 대부분의 값이 0이라서 "희소(Sparse)"하다.

```
입력: "ECACA §7.A Clearing Broker 역할 정의"
                    │
                    ▼
벡터 길이 = 전체 어휘 수 (수만 개)

인덱스:   [ ...  ECACA  §7.A  Clearing  Broker  역할  정의  마케팅  예산  ... ]
가중치:   [ ...  0.82   0.91   0.73     0.68    0.45  0.31   0      0   ... ]
                  ↑       ↑
            이 단어들이 높은 가중치를 받음
            나머지 수만 개는 전부 0
```

### BM25와의 차이

전통적인 BM25도 키워드 기반이지만, BGE-M3의 Sparse는 **학습된 가중치**를 사용한다.

```
BM25 (통계 기반):
  - 단어 빈도(TF)와 역문서 빈도(IDF)로 가중치 계산
  - 학습 없음. 단순 통계
  - "Clearing"이 많이 나오면 중요하다고 판단

BGE-M3 Sparse (학습 기반):
  - 신경망이 "이 단어가 검색에서 얼마나 중요한지" 학습
  - 문맥을 고려한 가중치 부여
  - "Clearing"이 금융 문맥에서 나올 때와 청소 문맥에서 나올 때
    다른 가중치를 줄 수 있음
```

### 강점

| 강점 | 예시 |
|---|---|
| **정확한 키워드 매칭** | "§7.A" 검색 → §7.A가 포함된 문서를 정확히 찾음 |
| **고유명사/ID** | "DEC-2026-0001", "ECACA" → 해당 문자열이 있는 문서만 반환 |
| **해석 가능** | 어떤 단어가 매칭됐는지 가중치로 확인 가능 (Dense는 블랙박스) |

### 약점

| 약점 | 예시 |
|---|---|
| **동의어 인식 못 함** | "해지" 검색 → "계약 종료" 문서를 못 찾음 (단어가 다르니까) |
| **다국어 교차 안 됨** | "clearing" 검색 → "청산" 한국어 문서 못 찾음 |
| **의도 파악 불가** | "IBKR이 우리를 어떻게 보고 있어?" → 키워드가 모호해서 결과 부정확 |

## 3. Multi-vector Retrieval (ColBERT 방식)

### 원리

Dense는 문장 전체를 벡터 **1개**로 압축한다. Multi-vector는 **토큰 하나하나마다 별도 벡터**를 생성한다.

```
Dense:      "ECACA §7.A Clearing Broker 역할"
                         │
                         ▼
                    벡터 1개 (1024차원)


Multi-vec:  "ECACA  §7.A  Clearing  Broker  역할"
               │      │       │        │       │
               ▼      ▼       ▼        ▼       ▼
            벡터1   벡터2   벡터3    벡터4    벡터5
           (1024)  (1024)  (1024)  (1024)  (1024)
```

검색 시 질문의 각 토큰 벡터와 문서의 각 토큰 벡터를 **교차 비교**한다.

```
질문: "§7.A에서 Clearing Broker를 어떻게 정의하나"

      질문 토큰      문서 토큰 (ECACA §7.A 조항 원문)
      ┌──────┐      ┌───────────────────────────┐
      │ §7.A │─────→│ §7.A  ← 최대 유사도 0.98   │
      │Clear.│─────→│ Clearing ← 최대 유사도 0.95 │
      │Broker│─────→│ Broker ← 최대 유사도 0.97   │
      │정의  │─────→│ defines ← 최대 유사도 0.82  │
      └──────┘      └───────────────────────────┘

      최종 점수 = 각 질문 토큰의 최대 유사도 합산
                = 0.98 + 0.95 + 0.97 + 0.82 = 3.72
```

### 강점

| 강점 | 예시 |
|---|---|
| **부분 매칭에 강함** | 긴 문서에서 특정 구절 하나만 관련 있어도 정확히 찾음 |
| **Dense + Sparse의 장점 결합** | 의미도 잡고 키워드도 잡음 |
| **정밀도 최고** | 세 방식 중 가장 정확한 매칭 |

### 약점

| 약점 | 예시 |
|---|---|
| **저장 공간 큼** | 토큰 수 × 1024차원. Dense 대비 수십~수백 배 저장 필요 |
| **검색 속도 느림** | 토큰 단위 교차 비교 → 대량 문서에서 시간 소요 |
| **인프라 복잡** | Qdrant에서 지원하지만 설정이 추가로 필요 |

## 4. 3가지 방식 종합 비교

```
              정확한 키워드       의미 파악        부분 매칭       저장 공간     속도
              ("§7.A")         ("계약 해지")     (긴 문서 내)
 ─────────────────────────────────────────────────────────────────────────
 Dense          ✗ 약함           ✓ 강함           △ 보통          적음        빠름
 Sparse         ✓ 강함           ✗ 약함           ✗ 약함          적음        빠름
 Multi-vec      ✓ 강함           ✓ 강함           ✓ 강함          많음        느림
```

| | Dense | Sparse | Multi-vector |
|---|---|---|---|
| **비유** | 책 전체를 읽고 "이 책은 계약법에 관한 책이야"라고 요약 | 책의 색인(Index)에서 키워드를 찾음 | 책의 모든 문장을 하나씩 대조하며 관련 구절을 찾음 |
| **벡터 수** | 문서당 **1개** | 문서당 **1개** (희소) | 문서당 **토큰 수만큼** |
| **잘 찾는 질문** | "clearing 관련 계약 내용 알려줘" | "§7.A" "ECACA" "DEC-2026-0001" | "§7.A에서 clearing broker의 의무를 구체적으로 어떻게 정의하는가" |
| **사용 시점** | 1차 후보 선정 (넓게) | 정확한 용어/ID 검색 | 최종 정밀 매칭 (좁게) |

## 5. 실전 적용: 3가지를 어떻게 조합하는가

문서 수가 수천~수만 건일 때 Multi-vector로 전부 검색하면 느리다. 단계적으로 범위를 좁혀가는 **퍼널 구조**가 현실적이다.

```
질문: "ECACA §7.A에서 Clearing Broker의 의무는?"

Stage 1: Dense + Sparse (빠르게, 넓게)
│
│  Dense  → 의미적으로 관련된 문서 50개 추출
│  Sparse → "ECACA", "§7.A" 키워드 매칭 문서 50개 추출
│  RRF 합산 → 상위 20개 선별
│
▼
Stage 2: Reranker (정밀하게, 좁게)
│
│  Cross-encoder로 질문-청크 쌍 재평가
│  상위 20개 → 상위 5개
│
│  (※ Multi-vector를 Reranker 대신 또는 함께 사용 가능.
│     다만 bge-reranker가 실용적으로 더 가볍고 빠름)
│
▼
Stage 3: Context Assembly
│
│  상위 5개 청크의 앞뒤 문맥 확장
│  출처 메타데이터 첨부
│
▼
최종 결과 반환
```

## 6. 이 프로젝트에 권장하는 조합

| 단계 | 사용 방식 | 이유 |
|---|---|---|
| **1차 검색** | Dense + Sparse | BGE-M3 하나로 두 벡터를 동시에 생성. Qdrant에서 Hybrid Search 내장 지원 |
| **2차 정밀화** | Reranker (`bge-reranker-v2-m3`) | Multi-vector보다 저장 공간 적고 구현 단순. 정확도는 충분 |
| **Multi-vector** | 당장은 미사용 | 문서 수가 수만 건을 넘거나, Reranker만으로 정확도가 부족할 때 도입 검토 |

**이유:** Multi-vector는 정확도가 가장 높지만, 저장 공간과 검색 속도의 트레이드오프가 크다. 4명 팀이 IBKR 프로젝트 수천 건으로 시작하는 단계에서는 **Dense + Sparse + Reranker** 조합이 복잡도 대비 최적이다.
